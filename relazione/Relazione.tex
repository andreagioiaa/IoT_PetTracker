\begin{abstract}
    Il progetto Pet-Tracker nasce con l'obiettivo di realizzare un sistema di localizzazione GPS in tempo reale per animali domestici di media taglia, 
    in grado di trasmettere la posizione, lo stato della batteria ed il numero di passi verso un server remoto accessibile tramite internet, 
    e di visualizzare tali dati attraverso un'applicazione dedicata.
\end{abstract}

\section{Architettura del sistema}
L'architettura del sistema Pet-Tracker segue il \textbf{Modello a 7 livelli dell'IoT World Forum (IoTWF)}, garantendo una transizione fluida dal dato fisico all'informazione per l'utente.

\subsection{Mappatura sul modello IoTWF}
\begin{itemize}
    \item \textbf{Livello 1 (Physical Devices):} Rappresentato dalla board LilyGo e dai sensori (GPS, Accelerometro).
    \item \textbf{Livello 2 (Connectivity):} Gestito tramite il modem LTE e protocolli di trasporto sicuri.
    \item \textbf{Livello 3 (Edge Computing):} Elaborazione locale sul microcontrollore per il filtraggio delle stringhe NMEA e il calcolo dei passi.
    \item \textbf{Livello 4 (Data Accumulation):} Persistenza dei dati sul server PocketBase (database SQLite).
    \item \textbf{Livello 5 (Data Abstraction):} Esposizione dei dati tramite API REST strutturate in formato JSON.
    \item \textbf{Livello 6 (Application):} L'applicazione mobile dedicata che trasforma le coordinate in posizioni su mappa.
    \item \textbf{Livello 7 (Collaboration and Processes):} L'interazione del proprietario con il sistema per il monitoraggio dell'animale.
\end{itemize}

\section{Hardware}
\subsection{LiLyGo T-SIM7670G-S3}
Il dispositivo utilizza una board LiLyGo T-SIM7670G-S3 \cite{lilygo_specs} che si basa sul modulo \textbf{ESP32-S3} dual-core, che gestisce sia la logica applicativa che la comunicazione con il modem SIM7670G tramite interfaccia UART.
\subsection{Antenna GPS e LTE}
Il sistema utilizza un'antenna flessibile LTE \cite{LTE} per la connettività dati e un'antenna ceramica attiva \cite{GPS} per il ricevitore GNSS integrato, essenziale per la precisione della localizzazione outdoor.
\subsection{Batteria e Power Management}
Viene impiegata una batteria ricaricabile 3000mAh 30A Litio-Ion ad alta capacità. Il monitoraggio avviene tramite un partitore di tensione collegato al pin \texttt{ADC\_BAT} (GPIO 4), permettendo di calcolare la percentuale di carica residua.
\subsection{BMA400 CJMCU-400}
L'accelerometro usato è un \textbf{BMA400} \cite{bma400} integrato nella breadboard \textbf{CJMCU-400}, che viene interrogato per rilevare il movimento, consentendo l'implementazione di un contapassi utile al monitoraggio della salute dell'animale.

\section{Firmware: Logica di funzionamento}
Il firmware, sviluppato in ambiente \texttt{Arduino} per il SoC \texttt{ESP32-S3}, gestisce l'acquisizione dei dati dai sensori, il controllo del modem LTE e l'ottimizzazione energetica tramite una macchina a stati finiti.

\subsection{Gestione della Persistenza e Hot Start}
Per ridurre drasticamente il \textit{Time To First Fix} (TTFF) del modulo GNSS, il firmware sfrutta la memoria \textbf{RTC (Real-Time Clock)} dell'ESP32. Questa memoria permette di mantenere i dati vitali anche durante i cicli di \textit{Deep Sleep}. 
Le variabili critiche, come l'ultima posizione valida (\texttt{lastLat}, \texttt{lastLon}) e i timestamp (\texttt{lastGpsDate}, \texttt{lastGpsTime}), vengono dichiarate con l'attributo \texttt{RTC\_DATA\_ATTR}.

Al risveglio del dispositivo, la funzione \texttt{iniettaGps()} esegue le seguenti operazioni:
\begin{itemize}
    \item Recupera le coordinate geografiche e temporali salvate nel ciclo precedente.
    \item Inietta la posizione nota nel modulo GNSS tramite il comando \texttt{AT+CGNSSPOS}.
    \item Sincronizza l'orario del modem con il comando \texttt{AT+CGNSSTIME}.
\end{itemize}
Questo meccanismo abilita la modalità \textbf{Hot Start}, che minimizza il tempo di ricerca dei satelliti e, di conseguenza, il consumo energetico.
\begin{lstlisting}[language=C++, caption={Ottimizzazione del fix GPS tramite iniezione di dati persistenti}]
void iniettaGps() {
  if (!hasGpsFix) return;
  // Costruzione comando per Hot Start basato su ultima posizione nota
  String cmdPos = "AT+CGNSSPOS=" + formatCoordinate(lastLat, true) + "," + latDir + "," + 
                  formatCoordinate(lastLon, false) + "," + lonDir + ",0,100";
  sendAT(cmdPos.c_str());
}
\end{lstlisting}

\subsection{Protocollo di Comunicazione HTTPS}
La trasmissione dei dati al backend PocketBase è affidata a una serie di comandi AT inviati via seriale al modem SIM7670G. Il firmware incapsula le informazioni in un oggetto JSON strutturato, inviato tramite una richiesta \texttt{HTTPS POST}. Il payload include:
\begin{itemize}
    \item \texttt{board\_id}: un identificativo univoco derivato dall'IMEI del modem;
    \item \texttt{geo}: un oggetto con latitudine e longitudine per la gestione cartografica;
    \item \texttt{battery} / \texttt{steps}: telemetria sullo stato di carica e sull'attività motoria rilevata;
    \item \texttt{sleep}: un flag booleano che segnala al server l'imminente entrata in stato di inattività.
\end{itemize}
\begin{lstlisting}[language=C++, caption={Esempio di strutturazione del payload JSON}]
void inviaDati(float l_lat, float l_lon, const BatInfo& bat, const String& timestamp, const StepData& step, bool isSleeping) {
  String json = "{";
  json += "\"timestamp\":\"" + timestamp + "\",";
  json += "\"geo\":{\"lon\":" + String(l_lon, 6) + ",\"lat\":" + String(l_lat, 6) + "},";
  json += "\"battery\":" + String(bat.voltage, 2) + ",";
  json += "\"sleep\":" + String(isSleeping ? "true" : "false");
  json += "}";
  // Invio tramite comandi AT HTTP
  modem.print(json);
}
\end{lstlisting}

\subsection{Risparmio Energetico e Deep Sleep}
Il sistema è progettato per massimizzare l'autonomia della batteria da 3000mAh. La logica di risparmio energetico è coordinata con l'accelerometro \textbf{BMA400}:
\begin{enumerate}
    \item \textbf{Rilevamento Attività}: Il firmware monitora costantemente il numero di passi. Ogni nuovo movimento resetta un timer di inattività (\texttt{lastActivityTime}).
    \item \textbf{Invio Finale}: Se non viene rilevata attività per un tempo superiore a \texttt{SLEEP\_TIMEOUT} (30 secondi), il sistema invia un ultimo pacchetto dati aggiornando il flag \texttt{sleep} a \texttt{true}.
    \item \textbf{Ibernazione}: Il microcontrollore entra in \textit{Deep Sleep}, spegnendo i moduli radio. Il risveglio è gestito tramite interrupt hardware sul pin \texttt{GPIO\_5}, generato dal BMA400 solo al rilevamento di un nuovo movimento.
\end{enumerate}
\begin{lstlisting}[language=C++, caption={Configurazione del Deep Sleep con risveglio tramite accelerometro}]
void enterDeepSleep() {
  Serial.println("\nEntro in Deep Sleep...");
  // Configura il pin di risveglio (GPIO 5) collegato al BMA400
  pinMode(WAKEUP_PIN, INPUT_PULLDOWN);
  esp_sleep_enable_ext0_wakeup(WAKEUP_PIN, 1);
  esp_deep_sleep_start();
}
\end{lstlisting}

\subsection{Monitoraggio Hardware e Batteria}
Il firmware gestisce in modo intelligente l'alimentazione per evitare letture errate o consumi parassiti:
\begin{itemize}
    \item \textbf{Controllo ADC}: Il circuito del partitore di tensione per la batteria viene attivato solo durante la lettura tramite il pin \texttt{BAT\_ADC\_EN}, garantendo la precisione del voltaggio acquisito dal pin \texttt{ADC\_BAT}.
    \item \textbf{Stato USB}: Attraverso la lettura dei registri hardware dell'ESP32, il sistema rileva se il dispositivo è collegato a una fonte di ricarica USB, settando dinamicamente il campo \texttt{charging} nel pacchetto dati.
\end{itemize}

\section{Back-end e Database}
Il centro nevralgico della gestione dati è basato su \textbf{PocketBase} \cite{pocketbase}, una soluzione open-source che integra un database embedded SQLite con un'interfaccia API REST. 

\subsection{Architettura di Hosting e Network Exposure}
L'istanza di PocketBase è stata ospitata su una workstation locale e data la natura del sistema IoT, che richiede la raggiungibilità del backend da parte del dispositivo (tramite rete LTE) e dell'applicazione mobile (tramite rete dati/Wi-Fi), si è reso necessario esporre il servizio locale su rete pubblica.
Il problema del \textit{NAT Traversal} e dell'assenza di un indirizzo IP pubblico statico sulla rete locale è stato risolto mediante l'impiego di \textbf{ngrok} \cite{ngrok}. Questa tecnologia di \textit{tunneling} crea un tunnel sicuro (HTTPS) tra l'endpoint pubblico di ngrok e la porta locale su cui è in esecuzione il server. 
Tale URL funge da \textit{Reverse Proxy}, reindirizzando il traffico HTTPS in ingresso verso l'istanza locale di PocketBase, garantendo così la persistenza dei dati e l'accessibilità dei servizi in tempo reale ovunque sia presente una connessione internet.

\subsection{Sistema di Backup e Data Retention}
Per garantire la resilienza dei dati e prevenire perdite accidentali dovute a guasti del server locale, è stato implementato un sistema di backup automatizzato. 
Il processo utilizza l'utility \textbf{rclone} \cite{rclone}, uno strumento a riga di comando per la gestione del cloud storage, configurato per eseguire la sincronizzazione della directory dei dati di PocketBase verso un bucket remoto su \textbf{Backblaze B2} \cite{backblaze}. 
L'automazione è gestita tramite un \textit{cron job} che esegue il backup in modo incrementale ogni primo giorno del mese.

\subsection{Monitoraggio e Business Continuity}
Per garantire l'affidabilità dell'intero ecosistema, è stato implementato un sistema di monitoraggio esterno tramite \textbf{UptimeRobot} \cite{uptimerobot}. Dato che l'accessibilità del backend dipende dalla stabilità del tunnel ngrok, il monitoraggio interroga l'endpoint HTTPS a intervalli regolari, inviando notifiche immediate in caso di downtime.

\subsection{Struttura dei dati}
I dati ricevuti vengono normalizzati e memorizzati in una collezione denominata \texttt{data\_sent\_raw}. La struttura del record prevede i seguenti campi:
\begin{itemize}
    \item \texttt{timestamp}: riferimento temporale dell'acquisizione.
    \item \texttt{lat} / \texttt{lon}: coordinate geografiche acquisite dal modulo GNSS;
    \item \texttt{battery}: stato di carica della batteria (espresso in voltaggio);
    \item \texttt{battery\_percent}: stato di carica della batteria (espresso in percentuale);
    \item \texttt{charging}: booleano che indica se la batteria è in corso di ricarica;
    \item \texttt{steps}: conteggio dei passi rilevato dall'accelerometro BMA400;
    \item \texttt{sleep}: booleano che indica se il dispositivo è in stato di "sleep" (assenza di movimento prolungata);
\end{itemize}

\subsection{Interfaccia API e API Rules}
L'accesso ai dati memorizzati nel backend avviene esclusivamente tramite un'interfaccia \textbf{RESTful API} fornita nativamente da PocketBase. Questo approccio permette di separare nettamente il livello di persistenza (Database) dal livello di fruizione (Applicazione Mobile).

\subsubsection{API Rules e Sicurezza}
Per garantire l'integrità e la riservatezza dei dati, sono state configurate specifiche \textbf{API Rules} basate su espressioni booleane:
\begin{itemize}
    \item \textbf{List/View:} Visualizzazione permessa solo agli utenti autenticati.
    \item \textbf{Create:} Abilitata per il dispositivo LilyGo tramite autenticazione dedicata.
    \item \textbf{Update/Delete:} Ristrette ai soli amministratori del sistema.
\end{itemize}

\subsection[Smistamento e Normalizzazione]{Logica di Smistamento e Normalizzazione (Server-side Hooks)}
Per ottimizzare la gestione dei dati e garantire la separazione delle responsabilità, è stata implementata una logica di smistamento automatico lato server tramite i \textit{JavaScript Event Hooks} di PocketBase. Questa architettura permette di mantenere la collezione \texttt{data\_sent\_raw} come un punto di accumulo ``grezzo'', delegando al server il compito di distribuire le informazioni nelle tabelle specializzate.

\subsubsection{Funzionamento del Trigger onRecordAfterCreateSuccess}
Il sistema sfrutta l'hook \texttt{onRecordAfterCreateSuccess}, che si attiva immediatamente dopo il salvataggio di un nuovo record inviato dalla board LilyGo. Il processo opera su tre direttrici principali:

\begin{itemize}
    \item \textbf{Persistenza Geografica ed Energetica:} I parametri di tensione, percentuale di carica e coordinate GPS vengono estratti e duplicati nelle collezioni \texttt{battery\_data} e \texttt{positions\_duplicate}. L'uso di blocchi \textit{try-catch} indipendenti garantisce che un errore in un sensore secondario non comprometta il tracciamento GPS.
    \item \textbf{Logica di Sessione (Activity Tracking):} Il trigger implementa una macchina a stati basata sul parametro \texttt{sleep}. Se il dispositivo è attivo, il sistema verifica l'esistenza di una sessione: in caso positivo aggiorna il conteggio passi, in caso negativo crea una nuova attività.
    \item \textbf{Gestione dello Stato Sleep:} Al ricezione di un pacchetto con \texttt{sleep: true}, il trigger chiude la sessione corrente impostando \texttt{is\_active = false}, permettendo una segmentazione precisa dei periodi di movimento.
\end{itemize}

\begin{lstlisting}[language=JavaScript, caption={Implementazione della logica di sessione in PocketBase}]
onRecordAfterCreateSuccess((e) => {
    const raw = e.record;
    const boardId = raw.getString("board_id");
    const timestamp = raw.getString("timestamp");
    const sleep = raw.getBool("sleep");
    const steps = raw.getInt("steps");

    // 1. Logica di gestione delle sessioni (Activity Tracking)
    if (!sleep) {
        if (activeActivity) {
            // Aggiorna passi e fine sessione corrente
            activeActivity.set("total_steps", currentSteps + steps);
            activeActivity.set("end_time", timestamp);
            e.app.save(activeActivity);
        } else {
            // Crea nuova sessione attiva se non esistente
            activitiesRecord.set("is_active", true);
            activitiesRecord.set("total_steps", steps);
            e.app.save(activitiesRecord);
        }
    } else if (activeActivity) {
        // Chiusura sessione per stato SLEEP del dispositivo
        activeActivity.set("is_active", false);
        e.app.save(activeActivity);
    }
    e.next();
}, "data_sent_raw");
\end{lstlisting}

\subsubsection{Vantaggi Architetturali e Corrispondenza IoTWF}
Questa implementazione riflette i principi del Modello IoTWF in due punti chiave:
\begin{enumerate}
    \item \textbf{Efficienza del Livello 3 (Edge):} La LilyGo invia un unico pacchetto ``bulk'', riducendo il tempo di attività del modem LTE e preservando la batteria.
    \item \textbf{Ottimizzazione del Livello 5 (Data Abstraction):} L'applicazione Flutter non deve processare dati grezzi; interroga API già normalizzate dal trigger, garantendo una visualizzazione fluida della cronologia.
\end{enumerate}
